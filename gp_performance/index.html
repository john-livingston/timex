    GP Performance: tinygp vs celerite2 in JAX¶

    Summary¶

    tinygp's quasisep solver is algorithmically O(N) but has higher constant factors than celerite2's C backend. The gradient difference is dramatic: 9-18x slower for tinygp. celerite2.jax (with C XLA ops) works with JAX 0.9.2 after a one-line fix (lax.zeros_like_array → jnp.zeros_like in ops.py). The latest celerite2 PyPI release (0.3.2) is outdated; install from git HEAD for JAX 0.8+ support.

    Benchmark Results (Apple M-series CPU, JAX 0.9.2)¶

    Log-likelihood evaluation (value only)¶

    N celerite2 numpy (C) celerite2.jax (C+XLA) tinygp (pure JAX)
    100 0.006 ms 0.008 ms 0.067 ms
    500 0.009 ms 0.031 ms 0.083 ms
    1000 0.012 ms 0.044 ms 0.098 ms
    5000 0.038 ms 0.147 ms 0.232 ms

    Gradient computation (value + grad via jax.value_and_grad)¶

    N celerite2.jax tinygp Speedup
    100 0.027 ms 0.231 ms 8.6x
    500 0.052 ms 0.655 ms 12.7x
    1000 0.084 ms 1.222 ms 14.6x
    5000 0.321 ms 5.736 ms 17.9x

    Gradients are the critical metric — NUTS evaluates value+grad every leapfrog step. celerite2.jax is 9-18x faster than tinygp, with the advantage growing at larger N.

    End-to-end: timex (numpyro) vs timer (PyMC)¶

    Both using celerite2 Matern-3/2, same dataset (hip67522c-gp, N~140 per band × 4 bands), 2000 tune + 2000 draws × 2 chains:

    timer (PyMC) timex (numpyro)
    Wall time 125 s 320 s
    t0 ESS 453 2754
    ror ESS 3456 4253
    gp_log_scale ESS 352 732

    ESS per second (higher = better):

    Parameter timer ESS/s timex ESS/s Winner
    t0 3.6 8.6 timex 2.4x
    b 20.7 9.0 timer 2.3x
    ror 27.6 13.3 timer 2.1x
    gp_log_scale 2.8 2.3 timer 1.2x

    Timer is faster per step but numpyro's NUTS explores the t0 posterior more efficiently. For transit timing (where t0 ESS matters most), timex produces better samples despite slower wall time.

    JIT compilation¶

    • tinygp log_likelihood: 0.1-0.2s per unique N (one-time)
    • tinygp value+grad: 0.2-0.4s per unique N (one-time)
    • Not a bottleneck for long MCMC runs

    Why tinygp is Slower¶

    1. Pure JAX vs C: tinygp is implemented entirely in JAX/Python (via equinox). celerite2 uses compiled C++ (Eigen) via pybind11. The O(N) algorithm is the same (semiseparable/quasisep matrix factorization) but C has much lower per-operation overhead.

    2. Pytree overhead: tinygp uses equinox dataclasses (JAX pytrees) for kernel objects. Each GP evaluation traverses the pytree structure, which adds overhead compared to celerite2's flat C arrays.

    3. Gradient method: tinygp relies on JAX's automatic differentiation (reverse-mode AD through the entire forward pass). celerite2 has hand-written C reverse-mode kernels (factor_rev, solve_lower_rev, etc.) that are much more efficient.

    4. Not a scaling issue: Both are O(N). The constant factor is the problem. At TESS scale (N=20k-100k), tinygp would be 4-10x slower than celerite2 per eval, and ~50-100x slower including gradients.

    celerite2's Architecture¶

    Forward pass (C)¶

    • driver.factor(t, c, a, U, V) → (d, W, S) — Cholesky factorization, O(N)
    • driver.solve_lower(t, c, U, W, Y) → solution, O(N)
    • driver.norm(t, c, a, U, V, Y) → log determinant, O(N)

    Reverse pass (C)¶

    • backprop.factor_rev(t, c, a, U, V, d, W, S, bd, bW) → (bt, bc, ba, bU, bV)
    • backprop.solve_lower_rev(...) → gradients w.r.t. inputs
    • backprop.solve_upper_rev(...), backprop.matmul_*_rev(...)

    All reverse functions are in C++ (pybind11), stored in celerite2/backprop.cpython-*.so.

    Kernel parameterization¶

    Matern32Term(sigma, rho) maps to celerite coefficients (c, a, U, V) via: - w0 = sqrt(3) / rho - S0 = sigma^2 / w0 - Specific a, U, V matrices derived from SHO representation with eps parameter

    JAX layer¶

    The PyPI release (0.3.2) was broken with JAX 0.6+ (xla_client.ops removed). Fixed in git HEAD: - b3ea056 — fix xla_client.ops removal (JAX 0.8+) - 22c907a — fix lax.zeros_like_array removal (JAX 0.9+), PR #166

    Installation¶

    conda install eigen  # needed for C++ compilation
pip install git+https://github.com/exoplanet-dev/celerite2

    API (drop-in replacement for tinygp in numpyro)¶

    from celerite2.jax import GaussianProcess, terms

kernel = terms.Matern32Term(sigma=amp, rho=scale)
gp = GaussianProcess(kernel)
gp.compute(t, diag=diag)

# In numpyro model:
numpyro.sample("obs", gp.numpyro_dist(), obs=residuals)

# For predictions (outside model):
gp.predict(residuals)

    Recommendation¶

    Switch timex from tinygp to celerite2.jax for 9-18x faster gradients. The API is nearly identical. For TESS-scale data (N>10k) this is essential.
